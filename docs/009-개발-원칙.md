# Crypto Pattern Lab — 개발 원칙

## 1. 목적

이 문서는 팀과 에이전트가 함께 개발할 때 지켜야 할 기본 개발 원칙을 정리한다.

## 2. 언어 및 표현 규칙

- 커밋 메시지는 한글로 작성한다.
- 코드 주석은 한글로 작성한다.
- 문서는 기본적으로 Markdown으로 작성한다.
- 식별자명은 코드 관례상 영어를 사용하되, 설명은 한글로 한다.

## 3. 구현 우선순위 원칙

- 문서 → 골격 → 작은 구현 → 검증 → 확장 순서로 간다.
- 먼저 끝까지 흐르는 작은 경로를 만든다.
- 추상화는 실제 중복이 보일 때 강화한다.
- 아직 모르는 도메인은 문서로 먼저 정리한다.

## 4. 모듈 개발 원칙

- 한 모듈은 한 가지 주된 책임만 가진다.
- 실행 앱과 공통 라이브러리를 분리한다.
- 외부 API 연동 코드는 도메인 로직과 분리한다.
- 실험 코드가 운영 코드에 침범하지 않게 한다.

## 5. 인터페이스 설계 원칙

- 인터페이스는 작고 명확해야 한다.
- 구현보다 계약을 먼저 정한다.
- 예외/오류 모델도 인터페이스의 일부로 본다.
- 확장 포인트는 열되, 초기 구현은 단순하게 간다.

## 6. 데이터 처리 원칙

- 원본 데이터를 가능한 한 보존한다.
- 정규화 규칙은 문서화한다.
- 시간 관련 필드는 의미를 분리한다.
- 중복 제거 키를 명시적으로 둔다.
- 재처리 가능성을 항상 염두에 둔다.

## 7. 테스트 원칙

- 핵심 매핑 로직은 테스트한다.
- 심볼 정규화는 테스트 우선순위가 높다.
- 스키마 변환은 샘플 payload 기반 테스트를 만든다.
- 네트워크 연동은 단위 테스트와 분리한다.

## 8. 설정 원칙

- 민감정보는 코드에 넣지 않는다.
- 예제 설정은 `.env.example`에 둔다.
- 로컬 설정과 운영 설정을 분리한다.
- 설정 이름은 일관되게 유지한다.

## 9. 문서화 원칙

새로운 결정이 생기면 아래 중 하나는 반드시 갱신한다.
- README
- docs 내 설계 문서
- 샘플 설정 파일

문서 없는 구현은 빠르게 부채가 된다.

## 10. Git 원칙

- 작은 단위로 자주 커밋한다.
- 커밋 메시지는 의도를 드러내게 쓴다.
- 구조 변경과 기능 추가를 가능하면 분리한다.
- 의미 없는 대규모 포맷 변경은 피한다.

## 11. 에이전트 협업 원칙

- 먼저 안전한 구조를 만든다.
- 불확실하면 문서 초안을 먼저 만든다.
- 파괴적 변경 전에는 확인한다.
- 반복 작업은 자동화한다.
- 배운 것은 문서나 메모에 남긴다.

## 12. 현재 우선순위

현재는 아래 순서로 진행한다.
1. 프로젝트 골격 안정화
2. 공통 스키마 코드화
3. collector 인터페이스 코드화
4. Binance/Bybit prototype
5. Upbit 확장
6. Ethereum transfer collector
