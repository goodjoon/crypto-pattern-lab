# Crypto Pattern Lab — System Design v0

## 1. 개요

이 문서는 Crypto Pattern Lab의 첫 설계 초안이다. 목표는 코인 거래 및 온체인 데이터를 통합 수집하고, 패턴을 발견하고, 전략 가설을 검증할 수 있는 연구 플랫폼을 만드는 것이다.

최종 목표는 자동매매가 아니라, 자동매매로 이어질 수 있는 **검증 가능한 분석 시스템**을 구축하는 것이다.

## 2. 문제 정의

대부분의 자동매매 프로젝트는 다음 문제를 가진다.

- 전략 검증 없이 실행 엔진부터 만든다.
- 백테스트에 수수료/슬리피지/체결 현실이 빠진다.
- 데이터 소스가 흩어져 있고 정규화가 안 되어 있다.
- 연구 결과가 재현되지 않는다.
- 온체인 이벤트와 거래소 데이터를 분리해서 본다.

이 프로젝트는 위 문제를 피하는 것을 목표로 한다.

## 3. 목표

### 3.1 단기 목표

- 거래소 데이터와 온체인 데이터를 수집한다.
- 공통 스키마로 정규화한다.
- 패턴 탐지 후보를 빠르게 실험할 수 있게 한다.
- 전략 가설의 유효성을 빠르게 검증한다.
- 사람이 읽기 쉬운 리포트를 생성한다.

### 3.2 중기 목표

- 실시간 신호 탐지 파이프라인을 만든다.
- 페이퍼 트레이딩 환경을 붙인다.
- 전략 성과를 지속 모니터링한다.
- 리스크 룰을 내장한 실행 엔진의 기반을 만든다.

### 3.3 장기 목표

- 검증된 전략의 일부를 자동 실행 시스템으로 연결한다.
- 데이터/신호를 재사용 가능한 내부 자산으로 만든다.
- 향후 데이터 상품화나 서비스화 가능한 기반을 확보한다.

## 4. 핵심 유스케이스

1. 특정 거래소/마켓의 체결 패턴을 수집하고 반복 패턴이 있는지 본다.
2. 거래소 간 가격 차이를 추적해 아비트라지 가능성을 본다.
3. 온체인 대규모 자금 이동이 가격에 미치는 영향을 본다.
4. 특정 이벤트 발생 이후 일정 시간 동안의 수익/변동성 반응을 측정한다.
5. 생성된 신호를 백테스트 및 페이퍼 트레이딩으로 검증한다.

## 5. 패턴 후보 카테고리

### 5.1 거래소 미시구조 패턴

- 체결 방향 불균형
- 호가 잔량 비대칭
- 단기 유동성 공백
- 특정 시간대 변동성 군집
- 급격한 체결량 증가 후 평균회귀/추세 지속 여부

### 5.2 거래소 간 패턴

- 동일 자산의 거래소 간 가격 괴리
- 현물/선물 베이시스 변화
- 펀딩비 변동과 가격 반응
- 오픈이자 변화와 단기 방향성 관계

### 5.3 온체인 패턴

- 고래 지갑 이동
- 거래소 입출금 급증
- DEX 대규모 스왑
- 브리지 유입/유출 급증
- 신규 컨트랙트/토큰 이벤트 반응

### 5.4 하이브리드 패턴

- 온체인 이벤트 발생 후 CEX 가격 반응
- DEX 유동성 변화와 CEX 괴리 확대
- 거래소 미시구조와 스마트머니 이동의 결합 신호

## 6. 시스템 아키텍처 초안

전체 구조는 5개 레이어로 나눈다.

### 6.1 Data Source Layer

외부 데이터 소스:
- 중앙화 거래소 REST/WebSocket API
- 온체인 RPC / 인덱싱 소스 / 서드파티 데이터 API
- 메타데이터 소스(토큰, 마켓, 체인 정보)

### 6.2 Ingestion Layer

역할:
- 거래소 실시간 데이터 수집
- 온체인 이벤트 수집
- 수집 실패 재시도
- 원본 이벤트 저장
- 타임스탬프 정규화

구성 예시:
- Exchange Collector
- On-chain Collector
- Scheduler / Backfill Worker
- Raw Event Queue

### 6.3 Normalization Layer

역할:
- 거래소별 형식 차이를 공통 이벤트로 변환
- 심볼/마켓/체인/주소 표준화
- 데이터 품질 검증

공통 이벤트 예시:
- trade_event
- orderbook_snapshot
- funding_event
- open_interest_event
- token_transfer_event
- dex_swap_event
- exchange_flow_event

### 6.4 Research & Signal Layer

역할:
- 피처 생성
- 패턴 탐지
- 전략 가설 관리
- 백테스트 실행
- 성과 리포트 생성

구성 예시:
- Feature Builder
- Pattern Detector
- Strategy Backtester
- Signal Evaluator
- Report Generator

### 6.5 Execution-ready Layer

초기에는 실제 주문보다 실행 준비 레이어까지만 설계한다.

역할:
- 실시간 신호 출력
- 페이퍼 포지션 추적
- 리스크 룰 시뮬레이션
- 실행 엔진으로 연결 가능한 인터페이스 제공

## 9. 데이터 저장 전략

### 9.1 저장 원칙

- 원본(raw)과 정규화(normalized)를 분리 저장한다.
- 가능하면 이벤트 소실 없이 raw를 남긴다.
- 분석용 집계 테이블은 별도로 만든다.
- 이벤트 시간(event time)과 수집 시간(ingest time)을 분리 저장한다.

### 9.2 후보 저장소

- PostgreSQL/TimescaleDB: 시계열 및 운영 데이터
- Object storage/local files: raw dump, 대용량 백필 데이터
- Redis: 실시간 캐시/버퍼

## 10. 검증 방식

전략 검증은 최소 다음을 반영해야 한다.

- 거래 수수료
- 예상 슬리피지
- 최소 주문 단위
- 실제 체결 가능성
- 거래 지연(latency)
- 거래소별 rate limit
- 포지션/노출 한도

검증 단계:
1. 아이디어 탐색
2. 과거 데이터 백테스트
3. 워크포워드 검증
4. 페이퍼 트레이딩
5. 소액 실거래

## 10.1 초기 대상 거래소 및 체인

### 거래소

초기 연결 후보는 다음과 같다.

- 국내: Upbit, Coinone
- 해외: Binance, Bybit

이 조합의 장점:
- 국내/해외 가격 괴리 및 시장 구조 차이 비교 가능
- KRW 마켓과 USDT 기반 글로벌 마켓을 함께 연구 가능
- 거래소별 체결 패턴과 유동성 차이를 비교 가능

### 블록체인 범위

초기 온체인 범위는 다음과 같다.

- Ethereum mainnet
- ERC-20 토큰
- Ethereum L2 체인(예: Arbitrum, Optimism, Base 등은 후속 후보)

### 참고사항

사용자는 이더리움이 PoW이던 시기까지의 경험은 깊지만, 현재의 Ethereum PoS 전환 이후 생태계와 최신 코인/L2 지형은 다시 빠르게 업데이트할 필요가 있다. 따라서 초기 문서와 리서치 단계에서 최신 생태계 맵을 정리하는 작업을 병행한다.

## 11. MVP 제안

### 11.1 MVP 목표

한 달 내에 작은 연구 시스템을 만든다.

### 11.2 MVP 범위

- 거래소 1~2개 연결

- 리포트 자동 생성

### 11.3 MVP 산출물

- 수집기 실행 가능 상태
- 데이터 적재 확인
- 패턴 실험 코드
- 일간/주간 리포트
- 다음 단계 우선순위 목록

## 14. 문서 운영 방식

원본은 Markdown 문서로 Git 저장소에서 관리한다.

권장 이유:
- 변경 이력 추적 가능
- 코드와 함께 관리 가능
- 자동 생성/수정이 쉬움
- Notion 의존 없이 재현 가능

필요하면 Notion에는 아래만 옮긴다.
- 요약본
- 회의 메모
- 공유용 로드맵
- 의사결정 기록 요약

즉, **원본은 Markdown / 공유와 보기 편한 정리는 Notion** 전략을 추천한다.
