# Crypto Pattern Lab — 아키텍처 원칙

## 1. 목적

이 문서는 Crypto Pattern Lab의 아키텍처 설계 원칙을 정의한다.

이 프로젝트는 단순한 스크립트 모음이 아니라,
- 거래소 데이터 수집
- 온체인 이벤트 수집
- 정규화
- 연구/검증
- 리포트 및 이후 실행 시스템 확장
을 장기적으로 다뤄야 한다.

따라서 초기부터 구조적 일관성을 갖는 것이 중요하다.

## 2. 현재 채택 방향

이 프로젝트는 다음 패턴을 혼합해 사용한다.

- 모듈형 모놀리스 기반 출발
- 포트/어댑터(헥사고날) 스타일 인터페이스 분리
- 이벤트 중심 데이터 파이프라인
- 원본 데이터와 정규화 데이터 분리
- 운영계와 연구계의 관심사 분리

## 3. 왜 이 방향을 택하는가

초기 단계에서 곧바로 마이크로서비스로 가면 아래 문제가 생긴다.
- 배포/운영 복잡도 증가
- 경계가 안정되기 전에 서비스 분리가 과해짐
- 작은 팀에서 생산성 저하

반대로 모든 것을 한 덩어리로 만들면 아래 문제가 생긴다.
- 수집기/분석기/스키마/운영 API 경계가 무너짐
- 실험 코드와 운영 코드가 섞임
- 확장 시 리팩터링 비용 증가

그래서 초기에는 **모듈형 모놀리스**로 시작하고,
경계가 안정되면 이후 독립 배포 가능한 단위로 분리한다.

## 4. 핵심 원칙

### 4.1 경계를 먼저 설계한다
기능 구현 전에 아래 경계를 먼저 분리한다.
- 수집
- 정규화
- 저장
- 분석/연구
- 리포트/API

### 4.2 외부 연동은 포트 뒤에 둔다
거래소 API, RPC, 메시지 큐, DB, 파일 시스템은 모두 포트 뒤에 둔다.

즉, 핵심 도메인 로직은 다음에 직접 의존하지 않는다.
- 특정 거래소 SDK
- 특정 DB 드라이버
- 특정 메시지 브로커

### 4.3 원본을 보존한다
raw 이벤트는 최대한 보존한다.
정규화는 raw를 기반으로 반복 가능해야 한다.

### 4.4 유실보다 중복을 허용한다
실시간 데이터 수집에서는 정확한 한 번 처리보다,
복구 가능한 최소 한 번 처리 전략이 더 현실적이다.

### 4.5 시간 의미를 분리한다
최소한 아래 시간을 분리한다.
- event_time
- ingest_time
- 필요 시 processed_time

### 4.6 연구 코드와 운영 코드를 분리한다
실험/리서치 코드는 빠르게 변한다.
운영 collector와 스키마 코드는 더 보수적으로 관리한다.

### 4.7 작은 핵심부터 만든다
처음부터 모든 거래소, 모든 체인, 모든 이벤트를 넣지 않는다.
작은 범위를 끝까지 관통하는 구조를 우선 만든다.

## 5. 패키지/모듈 원칙

권장 모듈:
- apps/api
- apps/collector-exchange
- apps/collector-onchain
- apps/research-worker
- libs/common-schema
- libs/connectors

원칙:
- apps는 실행 단위
- libs는 재사용 단위
- 도메인 규칙은 libs 또는 독립 모듈에 둔다
- 외부 연동 코드는 connectors에 둔다

## 6. 데이터 파이프라인 원칙

### 6.1 단계 분리
- 수집
- raw 저장
- 정규화
- 파생 데이터 생성
- 분석/리포트

### 6.2 재처리 가능성 확보
정규화 로직이 바뀌면 raw 기준으로 재처리할 수 있어야 한다.

### 6.3 스키마 버전 관리
normalized 이벤트는 장기적으로 버전 필드를 둘 수 있어야 한다.

## 7. 장애 대응 원칙

- 네트워크 오류는 기본적으로 재시도 가능 오류로 본다.
- 거래소 rate limit는 정상 상황의 일부로 본다.
- 끊김 이후 재구독/재연결은 표준 동작이어야 한다.
- 체크포인트는 운영 핵심 데이터다.
- 경고와 관측 가능성을 초반부터 넣는다.

## 8. 저장소 선택 원칙

초기 기본값:
- PostgreSQL: 운영 메타데이터 + 정규화 데이터
- Redis: 캐시/간단한 상태/버퍼
- raw dump: 파일 또는 오브젝트 스토리지 확장 가능

원칙:
- 먼저 단순한 저장소로 시작한다.
- 실제 병목이 나타나기 전까지 과도한 분산 저장소를 넣지 않는다.

## 9. 코드 품질 원칙

- 주석보다 이름이 먼저다.
- 주석은 왜 그렇게 했는지를 설명할 때만 쓴다.
- 테스트 가능한 구조를 우선한다.
- 인터페이스는 작고 분명하게 유지한다.
- 설정과 코드의 책임을 섞지 않는다.

## 10. 확장 원칙

아래 조건이 충족되면 서비스 분리를 검토한다.
- collector 부하가 독립 배포를 요구함
- 팀/릴리스 주기가 다름
- 장애 격리가 실질적으로 필요함
- 저장/처리 요구가 크게 달라짐

그 전까지는 모듈 경계를 엄격히 지키는 모놀리스가 더 낫다.

## 11. 현재 설계 결론

현재 기준의 최적 선택은 다음과 같다.

- Java 기반 모듈형 모놀리스
- 포트/어댑터 구조 적용
- raw/normalized 이벤트 분리
- collector와 research worker 관심사 분리
- 작은 MVP를 관통하는 구조 우선
